KitchenSync Basics

This Section of the document will step through some of the portions of KitchenSync that you're most likely to encounter and will get you on your feet with the core functionality before looking more closely at the details.

Key Pieces: Synchronizer & Actions

There are two main elements to KitchenSync - the synchronizer and actions.

Synchronizer

At the heart of KS is a class called Synchronizer. The Synchronizer works like a metronome for the entire system. It dispatches the update events that help the actions decide what to do next. Generally, this update occurs once every frame.

It's very unlikely that you will need to access Synchronizer directly from your programs. KitchenSync takes care of everything behind the scenes. For now, all you need to know is that it's there.

Running the command KitchenSync.initialize() at the beginning of your program will start the Synchronizer. More on that in a bit.

Why use a special class instead of an ENTER_FRAME listener or a Timer?

Using Synchronizer provides a number of advantages over a generic event dispatcher.

  • The Synchronizer sends out a timestamp (based on getTimer()) to its clients. That time is standardized for each update so all animations and actions are synchronized using the same current time. If this wasn't used, animations might appear slightly out of time with each other.
  • Having an abstracted Synchronizer means you can swap out the method used to dispatch the events. For example, you could use the ENTER_FRAME event of the stage or use a Timer with a specific interval or possibly even hook it up to a Midi word clock. To learn more about SynchronizerCores check the documentation.
  • The centralized synchronizer allows you to stop and start the entire system from a single point. Synchronizer also tracks the current number of active clients, the number of cycles since it started, and the 'official time' of the system.

Actions

Almost everything else in KitchenSync is related to actions. An action is a general term for anything that can be sequenced and executed using KitchenSync. Actions can be played, paused, looped, grouped and have a duration and delay (A complete list of ways to use actions will be discussed later in Controlling Actions). The most obvious action is a tween but KS comes packed with actions that do all kinds of things including playing sounds, loading files, calling functions, and so forth. Also groups, like the KSSequenceGroup, are also, themselves, actions and have all the properties of actions. Groups are discussed in more detail in a later section. Most of the different types of actions will be discussed in this document but if you want to learn more about any of them, you can snoop around in the ASDoc reference.

Actions are defined as any class that implements the IAciton interface. Most actions (except notably the KSSimpleTween) also inherit from AbstractAction which defines a default implementation of actions and add a few additional helpful functions.

For sake of clarity, KitchenSync uses a special naming convention for actions. For the most part, any class that implements the IAction interface will begin with "KS" such as "KSTween". In other words, any class starting with KS is an action and can be started, stopped, etc.

About Times

All actions have two basic times associated with them, they are duration and delay.

Duration is the length of time that an action takes to complete from start to finish. This is a very important property when using tweens because it describes how long the tween should take. However, some other actions that execute instantaneously, like KSFunctions, will always have a duration of 0. Actions have a boolean property, isInstantaneous, that will automatically be set to true if the action doesn't have a duration.

Delay is the amount of time that the action waits after the start() command is called before executing. Setting delays can let you space out the execution of several actions started at the same time.

My First Action

This section will walk you through the steps to get set up to run KitchenSync.

Installing KitchenSync

There are two basic ways to use KS. You can use the source code or you can use the pre-compiled SWC file. Both are available from the project's website in the download section.

To use the source code, just copy the org folder containing the org.as3lib.kitchensync packages into the source path of your project.

To use the SWC file, copy the SWC file into your source library and make sure your FLA file or FlexBuilder project contains a reference to that library path. I recommend creating a folder called "libs" in your project folder and making that your library path.

Instructions for setting the source path and library path can be found here:

Flash CS4 - Source Path | Library Path

Flex Builder - Place the files in your "src" folder | For SWCs, look under "Using SWC Files in your Projects"

Initializing the System

The first step to using KitchenSync is to initialize the system. Near the beginning (i.e. in the constructor) of your application's main class, document class, or in the first frame of your Flash movie, add the following line. 

KitchenSync.initialize(this, "2.0");

That's it! The system is running.

About the arguments:

The first argument in the call can be any display object that's on the stage. In most cases, using this will suffice but if your program is being loaded from another SWF, you may have to call the initialization function in an Event.ADD_TO_STAGE handler.

The second argument is totally optional and refers to the current version of KitchenSync. It will simply serve as a reminder of what version you are using and will throw an error if the version of KS you're using isn't what you're expecting.

Tip:

If you see the following error:

Error: KitchenSync has not been initialized yet and actions will not receive updates. Please run KitchenSync.initialize() in the main function of your program (or in the first frame).

That means you forgot to call the KitchenSync.initialize() function.

About Synchronizer Cores

It is possible to swap out the method that Synchronizer uses for dispatching updates to its clients. In most cases, this is unnecessary, however, your particular project may have special needs in this department. You can chose the method used by initializing KS using the command KitchenSync.initializeWithCore() (instead of initialize) and passing an instance of type ISynchronizerCore.

The library provides sync cores for ENTER_FRAME, Timer, and setInterval and you can create your own by creating a class that implements ISynchronizerCore.

By default, KitchenSync uses EnterFrameCore which seems to have the best performance overall.

Keeping Track of Actions

You don't need to keep a reference to an action for it to continue running, you only need the reference if you want to control the action after it starts (with stop, start, pause, etc.). Therefore, the code: 

var action:KSTrace = new KSTrace ("Hello, world! Sorry I'm late.", 5000);
action.start();

can be replaced with: 

new KSTrace ("Hello, world! Sorry I'm late.", 5000).start();

It's also possible to combine the constructor and the start method into one line because start() returns a reference to the action. However, start() returns an object typed as IAction so you may need to cast the return value to match the class you're creating. 

var action:KSTrace = new KSTrace ("Hello, world! Sorry I'm late.", 5000).start() as KSTrace;

It should also be noted that actions generally will try to delete themselves when they finish when the autoDelete property is set to true. So you usually won't need to worry about an action continuing to run once it's finished.

My first Tween

You're no doubt eager to see something more exciting happen. Let's make our next action a tween. There are two classes that implement ITween in KitchenSync, KSTween and KSSimpleTween. I'll use the KSSimpleTween action for this demo since it's a bit more straight forward. The most common tween is to move an object from one place to another.

View source code

Running this code should animate a black square from the left side of the screen to the right side of the screen. But let's take a closer look at what's happening.

First, of course, we initialize the KitchenSync library. 

KitchenSync.initialize(this);

The next block simply creates a new Sprite object and draws a 25px black square and adds it to the stage. 

var sprite:Sprite = new Sprite();
sprite.graphics.beginFill(0);
sprite.graphics.drawRect(10, 10, 25, 25);
addChild(sprite);

In the final piece of code, we're both creating and starting our tween object. 

new KSSimpleTween (sprite, "x", 0, 400, 3000, 500, Cubic.easeInOut).start();

Let's look at the individual parameters.

  • The first two, sprite and "x", are collectively the target of the tween. In this case, the x property of the sprite object.
  • The next two parameters, 0 and 400, represent the start and end values of the tween. In this case, the tween will move the x position of the sprite object from 0 to 400 pixels.
  • The next two parameters, 3000 and 500, are the duration and delay times in milliseconds. In other words, this animation will take 3 seconds and will start one-half second after start() is called.
  • The final parameter is the easing function that will be used to interpolate the values between the start and end. In other words, the shape of the motion. More on easing functions later on. This parameter should be a function without the parentheses ().
  • If you're using an IDE with code completion, you may notice two additional parameters not shown here, easingMod1 and easingMod2. These are arbitrary values that can be passed on to some easing functions such as Elastic or Oscillate to adjust their shape. See the easing function documentation for details. For now, they can be ignored.

You can use KSSimpleTween to tween any numeric value and for most cases, this will do the job. However, KSSimpleTween is called "simple" because it lacks some of the bells and whistles that KSTween has like jumping to a specific time in the tween or the ability to tween complex values like filters and sound transforms. On the other hand, sacrificing those features makes it more lightweight and therefore faster. We'll cover different types of tweens in detail in a later section.

Other types of actions

Tweening is not the end to what you can do with KitchenSync. Here's a partial list of other actions you can use. Many of them show their worth when thought of in the context of a sequence of actions. Some of them will be discussed in greater detail later in this document.

  • KSFunction and KSAsyncFunction - Calling functions from actions is a huge part of KitchenSync. This is especially apparent when the actions are grouped together in a sequence. Regular functions and asynchronous functions (like waiting for a response from a server) are supported.
  • KSSound - Loads and plays an MP3 file. It's a very quick way to add sound effects to animations. As a bonus, your sound will have all of the controls of an action like pausing and seeking.
  • KSSetProperty - Some values, like the boolean 'visible', may need to be set as part of a sequence but needn't be tweened. This allows you to set a property instantly.
  • KSLoader - Used for loading files from a server. There is also a KSLoadQueue that makes it easy to load multiple files as a single action.
  • KSDispatchEvent - Dispatches an event of your choosing using an action. Useful to signal the completion of a complex operation.
  • KSActionController - Let's you control another action (ie. stop, start, pause, etc.) using an action. Useful when you might have multiple animations happening simultaneously that could affect each other.

Grouping actions into a sequence

In this portion of the introduction, we will look at groups of actions. There are a few different types of action groups but all of them represent a group of several actions within a single object. An action group is defined by the interface IActionGroup which extends IAction. In other words, groups of actions are themselves actions and have all the same properties. Since an action group contains actions, an action group can contain other action groups. This is an example of the Composite Pattern.

The two most important types of group are KSSequneceGroup which plays all of its child actions one after the other, and KSParallelGroup which plays all of its children at the same time.

Shortcut for Combining Group Types

Since its common to use parallel and sequenced groups together, there is a shortcut to add them to each other easily. In the constructor for KSSequenceGroup and KSParallel group, parameter actions grouped in an array will be substituted with the other type. For example: 

var sequence:IActionGroup = new KSSequenceGroup (
    tweenA,
    [ tween1, tween2 ],
    tweenB
);

is identical to writing: 

var sequence:IActionGroup = new KSSequenceGroup (
    tweenA,
    new KSParallelGroup ( tween1, tween2 ),
    tweenB
);

and the same would happen in reverse with KSParallelGroup. If an array of actions is passed as an argument to the KSParallelGroup, that array is converted into a KSSequenceGroup.

Digging Deeper

Now you have the barebones basics of KitchenSync, let's look in more detail at what all you can use this library for.

Controlling Actions

All actions in KitchenSync implement a common interface that allows loads of control over the action. This section will run through all of the properties and methods associated with actions. Remember, this applies to all types of tweens and action groups as well.

Action methods

start()

Obviously, this causes the action to start running. If you're wondering why you're action isn't running, there's a good chance you forgot to call this method.

Start also returns a reference to the action which makes it easy to use an action's constructor and start it in a single line.

stop() and reset()

Calling stop() stops the action. The reset() method stops the action but also may bring it back to the state it was in before you started it.

pause() and unpause()

Actions can be paused and unpaused. Calling start() has the same effect as unpause().

togglePause()

AbstractAction adds an additional method, togglePause() that pauses an unpaused action and unpauses a paused action.

Jumping to a specific time in the action

Actions that implement the IJumpableAction interface (most of them), can use these commands to jump to a specific time in the action's timeline.

jumpToTime()

Jumps to a specific time in the action For example, if you have a tween that lasts 3 seconds, jumpToTime(2000) would move to a point 2 seconds into the animation.

jumpByTime()

Jumps ahead in an action by a specified time. For example, if you have a tween that lasts 5 seconds and you call jumpByTime(1000) after it's been playing for 1 second, you will jump to 2 seconds in.

kill() and autoDelete

Using kill() will attempt to delete the action and cleanup references it has to other objects. If autoDelete is set to true for an action, the action will automatically call kill() when it completes.

clone()

Clone produces a reference to a deep copy of the action. That is, you get a brand new action with the same properties. States of the cloned object, like whether it is running or not, will be ignored. The clone() method is typed as an IAction so you may need to cast the result to match the class you're cloning.

Tweens have a more rich set of options for cloning. They include cloneWithTweenTarget(), cloneWithTargetProperty(), and cloneReversed().

Controlling Groups of actions

Most of the time, if a method is called upon an action group, the children of that group will also be affected. For example, if you pause a group, all of the children within it will be paused as well.

Looping actions using KSLooper

While there is no "looping" functionality built into actions, the same effect can be achieved with a special kind of KSSequenceGroup called KSLooper. To loop a tween 5 times, for example, you can use: 

var tween:ITween = new KSSimpleTween ( sprite, "x", 0, 400, 3000, 0, Cubic.easeInOut );
var looper:KSLooper = new KSLooper ( tween, 5, true );
looper.start();

The first parameter is the action to loop, the second is the number of loops. The third is optional and it sets whether you want to reuse the same action for each loop (resetting in between) or whether you want to create clones of the action. In most cases, you can leave the last parameter off or set it to true.

Controlling actions with actions using KSActionController

As mentioned briefly before, it's possible to call most of these actions from another action called KSActionController. This is usually only used when sequences of actions need to interact with other sequences of actions. For the available action controls, see the ActionControlCommand class.

Action properties

These are the properties available to all action classes.

duration and delay

As mentioned before, duration is the length of time it takes for the action to complete and the delay is the time that the action waits after the start() method was called before it executes. Some actions will not allow you to set duration because of their nature like KSFunction which executes in a single frame.

isInstantaneous

Actions that execute immediately when started will have an isInstantaneous value of true. This value is read-only.

isRunning and isPaused

When the action is running, the isRunning value will be true. When the action is paused, the isPaused value will be true. The action is still considered to be running when paused. Both of these values are read-only.

progress and runningTime

Actions that extend AbstractAction add these two read-only properties.

progress is an estimate of the percentage complete of an action. For example, one second into a two second tween, it will return 0.5.

runningTime is time in ms since the start of the action or 0 if the action isn't running. If the action is paused, the result will be the running time at which it was paused. Actions with 0 duration will always return 0.

Both of these values include delay as part of the total time.

description

An optional text description can be set for each action. This makes it easy to identify actions when there are several similar ones running at the same time. Also helpful for debugging purposes.

autoDelete

As mentioned before, when set to true, actions will try to delete themselves when they complete to save memory. Good if you use lots of anonymous actions and bad if you want to use the same action object over and over again. True by default.

KitchenSync Events

Actions dispatch the following actions. You can add event listeners directly to actions or to groups of actions.

  • ACTION_START - Dispatched when an action is started
  • ACTION_PAUSE and ACTION_UNPAUSE - Dispatched when an action is paused or unpaused
  • ACTION_COMPLETE - Dispatched when an action is completed
  • Group actions dispatch the following additional actions:
    • CHILD_ACTION_START - Dispatched by a group when one of its children is started
    • CHILD_ACTION_COMPLETE - Dispatched by a group when one of its children is completed
    • ACTION_COMPLETE - Dispatched by a group when ALL of its children are completed

Tweening

For most people, the most exciting part of KitchenSync is tweening (for me, it's the sequencing, but that's just one man's opinion). This section will explore KS's tweening abilities in depth.

Core concepts: Tweens, TweenTargets, and Easing

We've already covered the basics of KSSimpleTween but now we'll get into the more complicated, and more sophisticated, KSTween. The first step is to understand the three parts of a tween that work in harmony.

  • Tween - The KSTween object. This piece contains all the timing-related information and is the actual action that you will call start() on. A KSTween object has one or more Tween Targets.
  • Tween Target - A tween target is a bundle that wraps up a property or other value that can be tweened by a KSTween object. In most cases, this will be a property of an object (see TargetProperty class) but it could also be something more complex like a tween target that sets a property of a soundTransform object or a property of a filter (which can't be set using a normal tween). The tween target will have a start and end value as well as providing an accessor to the current value of the target. Tween targets implement the ITweenTarget interface.
  • Easing Function - An easing function interpolates values between the start and end points of a tween. This determines the "shape" of the animation. Valid functions can be found in the org.as3lib.kitchensync.easing package.

Generally speaking, a KSTween will handle anything related to the timing of the tween while the ITweenTarget handles anything related to the values of the tween (a possible exception to the rule would be the easing function which is a property of KSTween). For example, KSTween holds the duration of the tween and the controls for starting, stopping and pausing while the tween target has the start and end values and holds a reference to the object to be tweened.

Tween targets are useful because they allow the value-setting portion of a tween to be abstracted away and separated from the timing and easing. This allows a variety of different types of values, such as filter properties, colors, and sound panning, to be set without subclassing KSTween for every instance. It also allows for a single KSTween object to control multiple tween targets.

Example of tweening position with KSTween and TargetProperty: 

var tweenX:ITweenTarget = new TargetProperty( sprite, "x", 0, 400);
var tweenY:ITweenTarget = new TargetProperty( sprite, "y", 0, 200);
var tween:KSTween = new KSTween ( [tweenX, tweenY], "2s", 0, Quartic.easeOut );
tween.start();
View Source

This tween creates two TargetProperty objects to control the x and y properties of a sprite then adds them to a new KSTween. You should see the sprite slide from (0,0) to (400,200) over two seconds. The Quartic.easeOut easing function gives it a smooth ending to the animation.

Notice that you can use several tween targets in a single KSTween.

This method for creating tweens can be a little wordy, but fortunately, there's a TweenFactory that can assist you.

Using the TweenFactory

The tween factory is a collection of methods used for creating tweens (Factory Method design pattern). This is the recommended way to create KSTweens. There are several functions, for a complete listing look at the documentation for TweenFactory. Here are some notable ones:

newTween() - Creates a new tween with one or more targets and one or more properties. Works pretty much the same as the KSSimpleTween constructor. The tween targets created will be of type TargetProperty. You can also provide an array of targets or properties if you want to tween more than one target/property at a time. 

TweenFactory.newTween(targets:*, properties:*, startValue:Number, endValue:Function, duration:* = 0, delay:* = 0, easingFunction:* = null):KSTween

newFilterTween - Creates a new tween on a property of a filter (such as the blurX property of a DropShadowFilter). 

TweenFactory.newFilterTween(target:DisplayObject, filter:Class, filterProperty:String, startValue:Number, endValue:Number, duration:Function = 0, delay:* = 0, easingFunction:* = null):KSTween

newScaleTween - Creates a tween that scales x and y simultaneously. 

TweenFactory.newScaleTween(target:DisplayObject, startValue:Number, endValue:Number, duration:Function = 0, delay:* = 0, easingFunciton:* = null):KSTween

newTimelineTween - Creates a tween that uses a TimelineController tween target. This actually tweens between frames in a Movie Clip. 

TweenFactory.newTimelineTween(target:MovieClip, startFrame:Object = null, endFrame:Object = null, duration:Function = -1, easingFunction:* = null):ITween

newTintTween - Creates a tween that changes a display object's color. 

TweenFactory.newTintTween(target:DisplayObject, startColor:Number = currentTint, endColor:Function = noTint, tintOpacity:* = 1.0, duration:* = 0, delay:* = 0, easingFunction:* = null):ITween

Using the object parser

If you've ever used Tweener, TweenLite or FuseKit, you've probably used a syntax similar to the KitchenSync object parser. This parser, accessed with the newTweenFromObject() method, allows you to pass an anonymous, arbitrary object as a parameter to create a new tween. This allows you to pick and choose exactly what parameters you wish to define. It also allows you to create tweens on several properties at once very easily.

For example, you could create a tween that tweens a sprite from (0,0) to (200, 200) and scales from 100% to 200% over 2.5 seconds using the following line. 

TweenFactory.newTweenFromObject({target: sprite, x:"0~200", y:"0~200", 
                                 scaleXFrom:1, scaleXTo:2,
                                 scaleYStart:1, scaleYEnd:2,
                                 easingFunction:Cubic.easeInOut,
                                 duration: "2.5s"
                                }).start();

For a complete list of keywords for the object parser see Appendix B.

Targeting multiple objects

In newTweenFromObject, the target can either be a single object or an array of objects to tween. So you could replace 'target: sprite' with 'targets: [circle, square, triangle]'.

Value ranges

As you can see from the example, there are three equally valid ways to give property ranges.

  1. Putting the values in quotes and using a tilde (~) allows you to set a range for a property.
  2. Using the property name plus From or _from and To or _to
  3. Using the property name plus Start or _start and End or _end

Using the latter two, if you omit one of the values (e.g. if you provide the end value with no start value) the value you omitted will be set to the current value of that property at the time of running the tween. Doing this is the same as using AUTO_TWEEN_VALUE (see below) for the omitted number.

Auto tween values

Say you want to tween a sprite along the x-axis from its current position at 100px to 500px. Normally, you would have to figure out the starting point manually which can be tedious under certain circumstances. Instead, you can use a special shortcut value, AUTO_TWEEN_VALUE, which automatically replaces missing values with the current value at the time that the tween starts. You can replace a start or end value with AUTO_TWEEN_VALUE. Here are some examples: 

// Tween from left to right then from right to left.
new KSSequenceGroup (
    TweenFactory.newTween (sprite, "x", 0, 400, "2s", 0, Cubic.easeInOut),
    TweenFactory.newTween (sprite, "x", AUTO_TWEEN_VALUE, 0, "2s", 0, Cubic.easeInOut)
).start();

// fade in - alpha from 0 to current value (1)
TweenFactory.newTween(sprite, "alpha", 0, AUTO_TWEEN_VALUE, "2s").start();
Importing AUTO_TWEEN_VALUE

AUTO_TWEEN_VALUE is actually a static constant (that evaluates to NaN) that is part of the org.as3lib.kitchensync.action.tween package. So if you want to use it, remember to import org.as3lib.kitchensync.action.tween.AUTO_TWEEN_VALUE.

When to use auto tween values

AUTO_TWEEN_VALUE can be used in KSTweens and KSSimpleTweens including all of the constructors in TweenFactory. When using TweenFactory.newTweenFromObject(), you can use AUTO_TWEEN_VALUE or just omit either the start or end value for a property.

Snapping values

In some cases you may want to make sure that tweens stick to whole number values, like if you want to make sure that text doesn't appear blurry. For this purpose there is a property of KSTween called snapToInteger. When true, the values will be rounded to the nearest integer. The default for this is set in KitchenSyncDefaults.

You can also make sure that after a tween is complete, it snaps to the end value (which with some easing functions, may not happen naturally) with snapToValueOnComplete. By default this is set to true.

Exploring other types of Tween Targets

One of the nice things about using ITweenTargets is that the targets can be used to set values that can't be set directly. For example, filter properties like blur are especially difficult since you have to replace a DisplayObject's entire array of filters to update even a single property. So far, we've primarily looked at one type of tween target, the TargetProperty, that sets values on a numeric property of an object. Here are some other commonly used tween targets. Many of these can be accessed using the factory methods in TweenFactory.

Filter tweens

Filter tweens target properties of BitmapFilter objects which manifest as visual effects on DisplayObjects. Filter tweens work a lot like TargetProperty tweens since most of them are built with a tween target called FilterTargetProperty. Instead of providing an object and a property, you provide a target movie clip, the name of the filter class, and one of the properties of that filter class. The constructor looks like this: 

FilterTargetProperty(target:DisplayObject, filterType:Class, property:String, startValue:Number, endValue:Number)

Color Tweens

There are a few types of tween targets that handle various color effects. They allow you to adjust the brightness, saturation, contrast, and tint of a display object. The first three, brightness, saturation, and contrast, use a color matrix filter to control the color. They all accept values between -1 and 1 where 0 is no adjustments. Unfortunately, you can only tween one of these values at a time (for now anyway). They all have similar constructors:

  • BrightnessTweenTarget(target:DisplayObject, startValue:Number, endValue:Number)
  • ContrastTweenTarget(target:DisplayObject, startValue:Number, endValue:Number)
  • SaturationTweenTarget(target:DisplayObject, startValue:Number, endValue:Number)

You can also tween tints, that is, tweening the color of a display object, there are a couple more options. 

TintTweenTarget(target:DisplayObject, startValue:* = CURRENT_TINT, endValue:* = NO_TINT, tintOpacity:Number = 1)

You can use a color in hex form (e.g. 0xFF3366) for the start and end values here. If you want to tween from the existing tint, use TintTweenTarget.CURRENT_TINT for the startValue. If you want to remove a tint, you can use TintTweenTarget.NO_TINT for the endValue. The last parameter, tintOpacity, adjusts the alpha of the tint.

Timeline tweens

Sometimes it is much simpler to create a complex animation using Flash's timeline than to attempt to reconstruct everything using ActionScript. However, mixing timeline animations and programatic animations can be tricky. That's why KitchenSync includes the awesome ability to run timeline animations using a KSTween object and a TimelineController tween target. Furthermore, you can pick any start or end frame (using frame numbers or labels) and adjust the duration of the tween to whatever you chose. You can even apply a custom easing function.

The simplest way to create timeline tweens is to use the TweenFactory.newTimelineTween() factory method. 

TweenFactory.newTimelineTween(target:MovieClip, startFrame:Object = null, endFrame:Object = null, duration:* = -1, easingFunction:Function = null, easingMod1:Number = NaN, easingMod2:Number = NaN):KSTween
Tip:

To play an animation backwards simply make the start frame later than the end frame. 

// tween from frame 100 backwards to frame 0
newTimelineTween(mc, 100, 0, TweenController.AUTO):KSTween;
Using natural duration

Most of these parameters are pretty straightforward. You provide a movie clip containing the animation, the start and end frame of the animation, and the details of the tween. You may notice that the duration is set to -1 by default. For this particular method, setting the duration to -1 (represented by the constant TimelineController.AUTO) will use the natural duration of the animation instead of a specified duration. For example, if you want to tween from frame 1 to frame 61 of your animation and your framerate is 30fps, setting duration to TimelineController.AUTO will make your tween the natural duration of the animation, in this case 2 seconds. This can be very helpful when you need your tween to sync up with other animations in the timeline.

Making sure you have enough frames

Timeline tweens work by stepping through each frame on the timeline meaning only frames that exist on the timeline will be shown and motion is not interpolated as it is in a regular tween. The result is that timeline tweens that alter the duration or easing function can appear a little bit chunky. For best results, make the animation much longer than you intend it to be then set the duration to your desired time. That way there are more frames for you to work with and it will appear smoother.

Groups

Let's take another look at groups. We've already seen the KSSequenceGroup and KSParallelGroup in action, now let's look in more depth at what you can do with these groups and at the other types of groups.

Adding and removing actions

The following methods are used for adding and removing actions from the group.

  • addAction()
  • addActionAtIndex()
  • removeAction()
  • removeActionAtIndex()

You can also get an array of the actions in the group with the childActions property.

Getting the total duration of the group

When working with tweens, it can be helpful to know how long a group will take to execute. Using the totalDuration property of a group will give you a fairly accurate estimate of the duration the entire group will take to complete.

Reversing child order

Simple, reverse the order of the child actions with reverseChildOrder().

Resetting children at the start of a group

Because when you use a group, you are starting several items at once. This can create some unexpected results, especially when using tweens, if the animations suddenly jump into a different place when the group runs. If you want the children of a group to reset when the group starts, you can set resetChildrenAtStart to true.

Using KSWait in sequences

If you need to pause momentarily between actions in a sequence, you can add an action called KSWait to the group. As you may have imagined, the Wait action does nothing except wait for a specified amount of time. 

// Wait time is the duration to wait.
var wait:KSWait = new KSWait(waitTime:*)
var group:IActionGroup = new KSSeuqenceGroup (
    doSomething(),
    wait,
    doSomethingElse()
);

Calling Functions using KS

KitchenSync also provides actions for calling functions. These may seem fairly boring at first, but when combined with KSSequenceGroups, they can be quite powerful. For example, you may have a site with multiple pages and when you click on a nav button, a sequence occurs where the current page animates out, the new page is initialized (with a function), and the new page animates in. It also is convenient to call several functions in a series in what's known as a command queue.

Types of Function actions

KitchenSync supports both synchronous (normal) and asynchronous functions with the action classes KSFunction and KSAsyncFunction. Both allow you to pass in parameters to the function and even retrieve results of the function.

Synchronous Functions

First, let's look at the KSFunction constructor: 

new KSFunction(func:Function, delay:Object = 0, ... args)

The last parameter, args, is an optional list of one or more arguments that will be passed to the function when it's called.

The function will wait for the specified delay time then call the function. If the function returns a result, the result will be stored in the KSFunction object's result property.

When providing the function name, make sure to leave off the () parentheses.

Asynchronous Functions

Asynchronous functions are functions that execute outside of a main thread of an application. Strictly speaking, Flash Player does not support threads but actions such as loading images from a server can still take place in the background while the program continues normally. Running asynchronous functions as actions is a little bit more complicated than regular functions but the core concepts are the same.

The constructor for KSAsyncFunction looks like this: 

new KSAsyncFunction(func:Function, completeEventDispatcher:IEventDispatcher, completeEventType:String, ... args)

The main difference here is that you need to define the conditions for the asynchronous function to be complete. In other words, you need to provide an event dispatcher and an event type to listen for that triggers when the function completes.

Loading files with KS

KitchenSync provides a set of actions that can be used for loading files from a server. They all extend ILoaderAction and mostly make use of the KSAsyncFunction. There are a few types of loader actions.

  • KSURLLoader - Use this class for loading text files and other generic data. It's analogous to the URLLoader class.
  • KSXMLLoader - Use this for loading XML files. Text results are automatically saved in the xml property as an XML object.
  • KSLoader - Use this in place of a Loader object for loading SWF files and images. When the load completes you can add the image to the stage by accessing the loader property.

Since usually the first thing you will want to do with the loaded files is get at their data, all of these classes have a property called resultList. This property can point to an array outside of the loader and when the loader completes, the data is automatically added to the result list. This makes the loaded data very easy to access. If you choose not to use the result list, you can still access the data from the data property of the loaders.

Load queues

Like with other actions, loaders become much more useful when sequenced. For this purpose, there is the KSLoadQueue. The load queue extends KSSequenceGroup. It takes a list of URLs and loads them in sequence. You can specify what type of ILoaderAction to use in the constructor. Sound actions

The KSSound action allows you an easy way to load and play a sound alone or as part of a sequence. Sequencing sounds together with other actions can be quite useful for when you want to play a sound effect as part of an event, or when you want to sync up sound and other actions (like showing subtitles for example).

It's also possible to tween the properties of the sound including pan and volume using a special tween target called SoundTransformTarget.

Playing sounds with KSSound

The KSSound action represents a flash.media.Sound object with all the benefits of actions including the ability to play, pause, and seek. To create a new KSSound, pass a Sound object in the constructor. You can also use a string url or URLRequest instead but be aware that the sound URL will load automatically when the action is created and may not be ready to play right away. When the KSSound action finishes playing, the action will be completed. Meanwhile, you can access the original Sound object and the SoundChannel object if you need more control. You can also set the offset when the sound begins playing. 

new KSSound(sound:*, delay:* = 0, soundOffset:* = 0)

Tweening sounds

You can also tween sound properties such as pan and volume using the SoundTransformTarget tween target. This class targets the soundTransform property of a sound (or any object with soundTranform like a NetStream). In the constructor, you would provide the object with the soundTransform property, the property of sound transform you want to access (e.g. "volume"), and the start and end values. There's another companion class, SoundTransformProperty, that enumerates the properties that you can tween.

The example below could be used to fade in a sound (from volume 0 to 1) over 5 seconds. 

var sound:Sound;
// load your sound here...

// fade up over 5 seconds
var fadeUp:ITweenTarget = new SoundTransformTarget(sound, SoundTransformProperty.VOLUME, 0, 1);
var tween:KSTween = TweenFactory.newTweenWithTargets(fadeUp, "5s");
tween.start();

Many properties, such as whether tweens snap to integers or what easing function to use, can be set here and will become the default choice of new actions. Appendix A: Time Strings

KitchenSync uses milliseconds as its standard unit of time. For example, setting a duration to an int value of 1500 will make the action run for 1,500 milliseconds or 1.5 seconds. However, in most cases, times can be set by providing string value such as "1.5 seconds". The system interprets the value using a class called TimeStringParser_en*.

Valid time strings can be almost anything. These are all acceptable formats:

  • "1 hour, 2 minutes, 3 seconds, 4 milliseconds"
  • "1h2m3s4ms"
  • "5sec,12fr"
  • "01:23:45;15" (timecode - equal to 1h, 23m, 45s, 15f)
  • ":03" (equal to 3s)
  • "300 frames"
  • "1.25s"
  • "5 milliseconds, 15mins, 6 hrs"
  • "0.25 days"

Notice these characteristics:

  • Abbreviations or full names can be used and plurals are optional. "second", "seconds", "sec", "secs", "s" are all valid ways to write "seconds".
  • Times can be separated by whitespace, commas, or by the beginning of the next number and order isn't important.
  • Decimals will be processed. Leading zeroes are optional.
  • Strings are recognized for milliseconds, seconds, minutes, hours, and days (God knows why you would need the last 2). Frames are supported two but they are less accurate than using real time.
  • Video timecode format (hh:mm:ss;ff) is recognized. Frames per second is interpreted based on a constant value set in the parser.

*: The 'en' represents that it understands English values. For localization or specialized purposes, the class used for parsing can be changed by providing a new implementation of ITimeStringParser in the KitchenSyncDefaults class.

: Frames are interpreted based on the framerate of the parser. They are not recommended because of their inaccuracy and should be considered deprecated. If you must use the frames option, please make sure you have set the frameRate to match the actual frame rate of the swf in the constructor.
